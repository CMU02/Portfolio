### SubHub

오프라인 우선 설계 + OCR 자동 입력 기반 구독 관리 앱 (Google Play 출시)

- 기간: 2025. 09. 30 - 2026. 01. 18
- 기술: React Native, Expo, OpenAI Vision, TypeScript, AsyncStorage
- 참여 인원: 1인 (앱 기획/개발), Kiro IDE Specs로 OCR 파이프라인·초기 UI 구현 초안을 빠르게 개발

1. OCR 모델 비교 및 기술적 의사결정

- 모델 벤치마킹: 영수증 인식을 통한 자동 입력을 구현하기 위해 동일한 24장의 영수증으로 여러 모델을 테스트했습니다.
  - 저사양 모델 (Llava-1.5: 4/24장, Uform: 0/24장): 토큰 제한 완화 시에도 실사용 수준의 인식 정확도 확보 실패.
  - 고성능 모델 (Llama-3.2-11B-Vision: 18/24장): 정확도는 우수하나, 모바일 앱 환경에서 비용과 토큰 사용량 과다하여 부적합하다고 판단.
- OpenAI Vision 도입: 정확도와 비용의 균형을 위해 OpenAI Vision으로 전환했습니다. gpt-4o-mini(22/24장)에서 gpt-4o(23/24장)로 전환하여 서비스명 오타·가격 자릿수 오류 케이스를 줄이고 인식 정확도 90% 이상을 확보했습니다.

2. 프롬프트 엔지니어링 및 데이터 효율화

- 토큰 최적화 (97% 절감): 프롬프트 구조 튜닝을 통해 초기 약 25,861개에 달하던 요청 토큰을 730~850개 수준으로 대폭 축소했습니다. 인식 정확도를 유지하면서 API 호출 비용과 응답 속도를 개선했습니다.
- 사용자 경험(UX) 개선: OCR 자동 추출 최적화를 통해 기존 9개였던 수동 입력 필드를 4개로 55% 줄여, 사용자 진입 장벽을 낮추고 등록 과정을 간소화했습니다.

3. 오프라인(Offline-First) 아키텍처

- 로컬 중심 설계: 초기 단계에서 무거운 클라우드 인프라에 의존하는 대신, AsyncStorage 기반의 오프라인 우선 구조로 설계했습니다. v1에서는 핵심 플로우에만 집중하고 푸시 알림·동기화는 v2로 연기하여 릴리즈를 약 2주 단축했습니다.
- 네트워크 독립성 확보: 네트워크 없이도 구독 조회/등록이 즉시 가능하며, 지하철 등 불안정한 환경에서도 핵심 기능이 정상 동작하여 사용자에게 기본 사용 경험을 보장했습니다.

### CleanBreath

안양시 금연/흡연 구역을 지도 폴리곤으로 시각화하는 웹 서비스

- 기간: 2024. 07. 26 - 2024. 11. 18 (초기 개발) / 2025. 11. 30 - 현재 진행 (유지보수)
- 기술: Java 21, Spring Boot, Spring Data JPA, MySQL, Next.js, KakaoMap SDK, TanStack Query, IndexedDB
- 참여 인원: 6인, PL/백엔드 담당 (BE: 2, FE: 4) — 일정 관리, 기술 의사결정, 코드 리뷰 주도 + 프론트엔드 유지보수 병행

1. 현장 답사 및 데이터 수집 툴 개발

- 공공 데이터 한계 극복: 공공데이터포털의 금연·흡연 구역 데이터는 안양시 전체 커버리지가 부족하고 경계 정보가 부정확하여 폴리곤 시각화에 바로 활용할 수 없었습니다.
- 자체 수집 툴 구축: 팀원 5명이 안양시 동안구를 3개 구역으로 나눠 약 1주일간 현장 답사를 진행했습니다. 좌표·메타데이터 입력용 수집 웹사이트와 CleanBreath DB 적재 API를 직접 개발하여, 공공 데이터와 현장 수집 데이터를 병합해 흡연구역 12개·금연구역 226개의 데이터 기반을 마련했습니다.

2. 쿼리 최적화 및 인덱스 튜닝

- 병목 분석: 폴리곤 렌더링 시 DB 응답 지연이 발생하여 EXPLAIN으로 쿼리 실행 계획을 분석한 결과, 위도/경도 조건에 인덱스가 누락되어 MySQL이 풀 스캔을 수행하고 있었습니다.
- 복합 인덱스 재설계: 위도/경도 복합 인덱스와 FK 인덱스를 재설계하고 쿼리를 리팩터링하여 일반 구역 조회 180ms → 23ms(87%↓), 아파트 구역 조회 40ms → 8ms(80%↓)를 달성했습니다.

3. 2중 캐싱 기반 데이터 효율화

- IndexedDB 선택 배경: 구역 데이터는 Key-Value 형태로 저장하기 적합한 정적 데이터입니다. Redis를 검토했으나 학생 신분으로 인프라를 지속 운영하기 어려워, 브라우저에 내장된 Key-Value DB인 IndexedDB를 대안으로 선택했습니다.
- 2중 캐싱 구조: IndexedDB(30일 보존)와 TanStack Query(staleTime 1시간)를 조합하여 반복 방문 시 API 호출 95%를 절감했습니다. 버전 기반 캐시 무효화로 데이터 정합성을 유지합니다.

4. UI 리뉴얼 및 렌더링 성능 최적화

- 상태 관리 개선: Zustand로 지도·UI·필터 상태를 분리하여 불필요한 리렌더링을 제거하고, React Compiler 도입으로 자동 메모이제이션을 적용했습니다.
- 성능 개선: Lighthouse 성능 점수 74 → 91점(+17점)을 달성했습니다. 유지보수 시에는 Kiro IDE 에이전틱 워크플로우(오류 재현 → 원인 분석 → 수정 → 검증)를 활용하여 문제를 해결하고 있습니다.
