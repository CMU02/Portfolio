# SubHub

1. 프로젝트 개요

- 오프라인 우선 설계 + OCR 자동 입력 기반 구독 관리 앱 (Google Play 출시)
- 기간: 2025. 09. 30 - 2026. 01. 18
- 기술: React Native, Expo, OpenAI Vision
- 참여인원: 1인 (앱 기획/개발)

## 성과

1. OCR 모델 비교 및 기술적 의사 결정

- 모델 벤치마킹 및 한계 돌파: 영수증 인식을 통한 자동 입력을 구현하기 위해 Cloudflare의 여러 모델을 테스트 했습니다.
  - 저사양 모델 (Llava-1.5, Uform 등): 토큰 제한 완화 시에도 실사용 수준의 인식 정확도 확보 실패.
  - 고성능 모델 (Llama-3.2-11b-vision): 정확도는 우수하나, 모바일 앱 환경에서 비용과 토큰 사용량 과다하여 부적합하다고 판단
- OpenAI Vision 도입: 정확도와 비용의 균형을 위해 OpenAI Vision으로 전환했습니다.초기 gpt-4o-mini에서 gpt-4o로 전환으로 영수증 인식 정확도 90% 이상 확보했습니다.

2. 프롬프트 엔지니어링 및 데이터 효율화

- 토큰 최적화 (97% 절감): 프롬프트 구조 튜닝을 위해 초기 약 25,861개에 달하던 요청 토큰을 700~850개 수준으로 대폭 축소했습니다.이를 통해 인식 정확도를 유지하면서 API 호출 비용과 응답 속도를 개선했습니다.

- 사용자 경험(UX) 개선: OCR 자동 추출 최적화를 통해 기존 9개였던 수동 입력 필드를 4개로 55%로 줄여, 사용자 진입 장벽을 낮추고 등록 과정을 간소화했습니다.

3. 오프라인(Offline-First) 아키텍처

- 로컬 중심 설계: 초기 단계에서 무거운 클라우드 인프라에 의존하는 대신, AsyncStorage 기반의 오프라인 우선 구조로 설계했습니다.

- 네크워크 독립성 확보: 네크워크 없이도 구독 조회/등록 즉시 가능하며, 지하철 등 불안정한 환경에서도 핵심 기능 정상 동작 함으로서 사용자에게 기본 사용 경험을 보장했습니다.

## 배경 및 문제 정의

### 서비스 배경

- 넷플릭스, 유튜브 프리미엄, 클라우드, 게임 패스 등 구독 서비스가 늘어날수록 사용자는 “내가 뭘 얼마나 쓰고 있는지”를 파악하기 어려워한다.

- 스프레드시트나 메모 앱으로 관리하려고 해도, 영수증마다 서비스명·가격·결제 주기·갱신일을 일일이 입력해야 하는 번거로움 때문에 금방 포기하게 된다.

> “구독은 계속 늘어나는데, 관리 도구는 귀찮아서 안 쓰게 되는” 상황이 SubHub의 출발점이었다.

### 핵심 문제

1. 구독 등록이 너무 귀찮다

- 영수증을 보고 서비스명·가격·주기·갱신일을 모두 타이핑해야 함.

2. 월 구독 지출이 한눈에 보이지 않는다

- 어떤 구독에 얼마를 쓰는지, 총합이 얼마인지 직관적으로 보기 어렵다.

3. 데이터 신뢰성과 접근성

- 네트워크가 불안정하거나 오프라인인 상태에서도 구독 정보를 확인하고 관리할 수 있어야 한다.

## 방안 도출 과정

### AI OCR 기반 구독 등록

1. 초기 접근:

영수증 사진만 찍으면 구독 정보가 자동으로 입력되도록 만들기 위해, 여러 Vision 모델을 벤치마킹한 뒤 최종적으로 OpenAI GPT‑4o Vision을 채택했다.

#### 추출 목록

- 서비스명 (serviceName)
- 가격 (price)
- 통화 (currency)
- 카테고리 (category)
- 다음 결제 예정일

2. Cloudflare 제공 모델 벤치마킹

- 먼저 비용과 인프라 단순성을 고려해 Cloudflare Workers + Cloudflare AI 모델 조합을 우선 검토
  - 저사양 모델 (예: Llava‑1.5, Uform 계열)
    - 장점: 비용 저렴, 토큰 제약 완화에 유리

    - 한계: 토큰 제한을 풀어도, 실제 영수증 환경(국문·복합 레이아웃 등)에서
      서비스명·금액·주기·다음 결제일을 안정적으로 추출할 정도의 인식 정확도에 도달하지 못함. <br>
      ⇒ 실사용 수준의 OCR 품질을 확보하지 못해 채택하지 않음.

  - 고성능 모델 (예: Llama‑3.2‑11B‑Vision)
    - 장점: 필드 인식 정확도는 상당히 우수

    - 한계:모바일 앱에서 자주 호출하기에는 비용과 토큰 사용량이 과도하며, 개인 사이드 프로젝트 규모에서 지속 운영하기에 단가와 사용량 예측이 부담. <br>
      ⇒ “정확도는 좋지만, SubHub의 비용 구조와 맞지 않는다”고 판단해 채택하지 않음.

> 이 과정에서 “정확도만 좋은 모델”이 아니라, <br>
> 정확도·토큰 효율·운영 비용을 함께 만족하는 모델이 필요하다는 기준이 명확해졌다.

3. OpenAI GPT‑4o Vision + 프롬프트 튜닝 (최종 채택)
   Cloudflare 모델 실험 후, OpenAI GPT‑4o Vision을 도입해 다시 설계했다.

이미지 한 장을 입력하면 서비스명, 가격, 통화, 카테고리, 다음 결제 예정일을
JSON 형태로 바로 받는 단일 단계 파이프라인을 구성했다.

### 오프라인 퍼스트 아키텍처 선택

구독 정보를 확인하는 상황은 지하철/비행기/외국 등 네트워크가 불안정한 환경에서도 자주 발생할 수 있다.
이 때문에 SubHub의 핵심 기능(구독 목록 조회, 기본 편집)은 완전 오프라인에서도 100% 동작해야 한다고 정의했다.

- 클라이언트 저장소로 AsyncStorage를 채택

- 구독 목록, 설정, 통화 정보 등 핵심 데이터를 모두 로컬에 유지

- 네트워크는 OCR 요청과 환율 조회, 광고 노출에만 사용

이렇게 설계함으로써:

- 서버 장애나 네트워크 불안정 시에도 구독 정보 접근은 항상 가능

- 백엔드 동기화가 없기 때문에 초기 릴리즈 범위와 구현 복잡도를 크게 줄일 수 있었다.

> 단, 이 선택은 뒤에서 이야기할 기기 변경 시 데이터 손실이라는 trade-off를 동반한다.

### 출시 범위 조정

초기 기획에는 결제 예정 알림, 알림 전략 튜닝까지 포함되어 있었으나, 이 모든 것을 한 번에 구현하면 릴리즈 일정이 2주 이상 지연될 것이 분명했다.

그래서 v1에서는 핵심 플로우에만 집중

1. AI OCR 기반 구독 등록
2. 구독 대시보드 및 지출 요약
3. 다국어/다크모드 지원

푸시/알림 및 고급 알림 전략은 v2이후로 연기

이 결정으로 약 2주 정도의 릴리즈 단축을 가져가면서, 먼저 MVP를 시장에 내보내 실제 사용성을 검증하는 전략을 취했다.

## 주요기능 구현과 결과

### AI 기반 구독 등록

초기에 Vision을 쓸 때는 프롬프트가 길어져 한 번 호출에 최대 25,889 토큰까지 사용되는 문제가 있었지만,

“영수증 전체 설명”이 아닌
“필요한 필드 4개만 뽑는 프롬프트”로 재설계하고

출력 포맷을 엄격한 JSON으로 제한한 결과,

호출당 토큰 사용량을 730~850 토큰 수준으로 안정화했다.

- 그 결과:
  - 사용자 입력 필드: 9개 → 4개로 축소
    (대부분을 AI가 채우고, 사용자는 최종 확인·수정만 수행)

  - Cloudflare 저사양 모델 대비 현실적인 정확도,
    고성능 모델 대비 낮은 비용·예측 가능한 토큰 사용량을 동시에 달성했다.

### 구독 관리 대시

앱 내 대시보드는 다음 정보를 한 화면에서 제공한다.

- 전체 월간 구독 지출 합계

- 카테고리별(스트리밍/클라우드/게임 등) 지출 합계

- 각 구독의 다음 결제일, 결제 주기

- 다중 통화 지원(예: 원화/달러/엔화)으로 해외 구독까지 확인 가능

### 오프라인 퍼스트 동작

- 모든 구독 데이터는 AsyncStorage에 저장되어 오프라인에서도 그대로 읽을 수 있다.

- 네트워크가 없어도:
  - 구독 목록 조회

  - 구독 수정/삭제

  - 기본 통계(월 총액, 카테고리 합계)
    가 모두 동작하도록 설계했다.

이는 사용자가 지하철, 비행기, 해외 로밍 불가 상태에서도 자신의 구독 상황을 확인할 수 있게 하는 데 초점을 둔 설계다.

## 결과 검증

SubHub는 아직 대규모 트래픽이 있는 서비스는 아니지만, 다음과 같은 관점으로 검증했다.

1. AI OCR 비용·효율성 검증

- 이전 방식 대비 토큰 사용량을 대폭 줄이면서, 인식 정확도와 속도를 동시에 확보.
- 이 구조로 향후 일주일 3회 무료 + 보상형 광고로 추가 OCR 모델을 운영할 여지가 생김.

2. UX 개선 효과

- 텍스트 수동 입력 위주의 기존 구독 관리 방식과 비교해, 이미지 촬영 → 자동 채움 흐름으로 입력 시간이 체감상 크게 단축.

- 입력 필드 수를 9개에서 4개로 줄임으로써, “AI가 채우고 사용자는 확인만 한다”는 느낌의 플로우를 완성.

3. 오프라인 퍼스트 검증

- 네트워크를 강제로 끊은 상태에서 구독 목록 조회/편집이 정상적으로 동작하는지 반복 테스트해, 핵심 기능이 네트워크 품질과 무관하게 동작함을 확인했다.

## 한계점 보완 방향

### 기기 변경 시 데이터 손실 (오프라인 퍼스트의 Trade-off)

현재 SubHub는 서버 동기화 없이 AsyncStorage만 사용하는 완전 오프라인 구조다.
이로 인해:

- 기기를 변경하거나 앱을 삭제하면 모든 구독 데이터가 사라진다.

- 사용자가 여러 기기(폰·태블릿)를 동시에 사용할 수 없다.

#### 보완 방향:

기본은 오프라인을 유지하되, 선택적으로 클라우드 백업 옵션을 제공하는 하이브리드 모델을 고려 중이다.

- 예: Supabase 또는 AWS 기반 간단한 백엔드로
  - “내 계정에 백업” 버튼을 누르면 암호화된 형태로 구독 리스트를 업로드

  - 새 기기에서 로그인 시 복원

- “오프라인만 쓰겠다”는 사용자는 기존처럼 완전 로컬 모드 유지.

### AI OCR 오인식에 대한 보정 UX 부족

AI가 잘못 인식한 경우 사용자가 쉽게 수정할 수 있는 UX가 필수지만, 현재는 이 부분이 충분히 다듬어져 있지 않다.

- 금액/통화, 서비스명, 결제 주기가 틀릴 경우 즉시 수정 가능해야 하고
- 자주 틀리는 패턴(특정 카드사 영수증, 특정 언어 포맷)은 별도의 규칙 보정이나 후처리가 필요하다.

#### 보완 방향:

- OCR 결과 화면에서 각 필드를 한 번에 편집 가능한 폼으로 제공하고

- “이 값은 항상 이렇게 바꾸기” 같은 사용자 정의 룰(간단한 로컬 룰)을 추가하는 기능 고려.

### 결제 라이프사이클 관리 미흡

- 현재는 결제 예정 알림 중심으로 설계되어 있고, 실제 결제가 완료되었는지, 구독을 해지했는지에 대한 흐름이 약하다.

#### 보완 방향:

- 구독별로 “결제 완료/미결제/해지” 상태를 표시할 수 있는 상태 머신 설계

- 해지된 구독을 별도 탭으로 보여줘 “내가 줄인 구독 비용”을 시각화

- 장기적으로는 카드사 알림/이메일 파싱까지 연계할 수 있는 확장 포인트로 설계
