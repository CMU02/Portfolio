# Cleanbreath

1. 프로젝트 개요

- 안양시 금연/흡연 구역을 지도 폴리곤으로 시각화하는 웹 서비스
- 기간: 2024. 07. 26 - 2024. 11. 18 (초기 개발) / 2025. 11. 30 - 현재 진행 (유지 보수)
- 기술: Java 21, Spring Boot 4.0, Spring Data JPA, MySQL, Next.js, KakaoMap SDK
- 참여 인원: 6인, PL/백엔드 담당 (BE: 2, FE: 4), PL로서 일정 관리, 기술 의사결정, 코드 리뷰 주도 + 프론트엔드 유지보수 병행

## 성과

1. 쿼리 최적화 및 인덱스 튜닝
   - 병목 지점 분석: 지도 위 폴리곤 렌더링 시 특정 주역 조회에서 발생하는 DB 응답 지연 문제를 로그와 실행 계획으로 분석했습니다.
     그 결과, 위도/경도 조건에 인덱스가 누락되어 풀 스캔이 발생하고 있음을 확인했습니다.
   - 성능 개선 결과: 위도/경도 복합 인덱스와 외래 키(FK) 인덱스를 추가하여 쿼리 효율을 높였습니다.
     이를 통해 일반 구역 조회의 평균 응답 속도를 기존 180ms에서 23ms로 단축했습니다.
2. 2중 캐싱 기반 데이터 효율화
   - 전략적 캐싱 설계: 금연/흡연 구역 데이터가 상대적으로 정적이라는 점에 착안하여,
     IndexedDB(30일 보존)와 TanstackQuery(staleTime 1시간)를 조합한 2중 캐싱 구조를 설계했습니다.
   - 데이터 정 합성 유지: 월말 배치 스케줄러를 통해 서버의 변경된 데이터만 차분 업데이트하고, 버전 정보를 기준으로 캐시 무효화를 처리했습니다.
     이를 통해 반복 방문시 불필요한 API 호출을 최소화했습니다.
3. UI 리뉴얼 및 렌더링 성능 최적화
   - 상태 관리 및 컴파일러 도입: 프론트엔드 유지보수를 위해 UI 구조를 리뉴얼 했습니다.
     Zustand를 활용한 효율적인 상태 관리와 React Compiler 도입을 통해 불필요한 리렌더링을 최소화했습니다.
   - Lighthouse 지표 개선: 최적화 결과, 리뉴얼 전 기준 74점이었던 Lighthouse 성능 점수를 91점까지 끌어올리며 초기 로딩 속도와 사용자 경험을 향상 시켰습니다.

## 배경 및 문제 정의

### 도메인 배경

안양시 곳곳에 금연·흡연 구역이 지정되어 있지만, 경계가 모호하거나 실제 현장과 표시가 맞지 않는 경우가 많았다.

- 시민 입장에서는 “여기가 금연인지, 옆 벤치는 되는지”를 알기 어렵고

- 단속이나 민원 과정에서 “여긴 금연구역이 아니다 / 맞다”를 두고 시비가 생기곤 했다.

공공데이터포털에서 제공하는 금연·흡연 구역 데이터만으로는 안양시 전체를 커버하기에 부족했고, 실제 현장 상황을 제대로 반영하지 못하고 있었다.

### 데이터 측면 문제

1. 공공 데이터 부족

- 제공된 데이터가 적고, 실제 현장과 불일치하는 구간이 다수.

2. 좌표·경계 정보의 부정확성

- “건물 단위” 정보는 있지만, 어디까지가 금연/흡연 구역인지 경계가 시각적으로 보이지 않음.

3. 대량 지도 데이터 조회 성능 문제

- 지도의 줌/이동에 따라 수백·수천 개의 구역 폴리곤을 불러와야 하는데,
  단순 쿼리로 접근하면 DB 풀 스캔이 발생해 응답 속도가 느려질 위험이 있었다

## 방안 도출 과정

### “현장 데이터 + 수집 툴” 전략

공공 데이터만으로는 부족하다고 판단해, 팀원들과 함께 직접 현장 답사 + 자체 데이터 수집 툴 전략을 선택했다.

- 안양시 동안구 내 3개 구역(2곳, 2곳, 1곳)으로 나누어 친구들이 직접 현장을 돌며:
  - 실제 흡연 부스 위치

  - 금연 표지판 위치

  - 시민들이 실제로 사용하는 흡연 장소
    등을 조사했다.

나는 친구들이 데이터를 쉽게 입력할 수 있도록:

- 좌표·메타데이터를 등록하는 수집용 웹사이트

- 이를 받는 백엔드 API
  를 구현했다.

> 친구들이 내가 만든 툴로 데이터를 모으고, 그 데이터가 다시 CleanBreath의 폴리곤으로 시각화되는 과정을 통해
> “좋은 백엔드/툴은 팀원들의 발로 뛴 노력을 잃지 않게 받쳐주는 장치”라는 점을 배웠다고 정리할 수 있다.

### 폴리곤 기반 시각화와 DB 설계

“여기가 금연인지 한눈에 보이도록” 만드는 것이 핵심 목표였기 때문에, 지도 위 폴리곤 렌더링을 중심으로 설계했다.

- 금연/흡연 구역을 위도·경도 좌표 배열로 저장해 폴리곤으로 렌더링

- DB에는 다음을 고려해 스키마를 설계:
  - 구역 ID / 타입(금연, 흡연) / 행정구역 / 좌표 리스트

  - 위경도 복합 인덱스로 특정 영역 조회를 빠르게 처리

  - 아파트 단지/일반 구역을 구분해 쿼리 패턴 최적화

### 성능 문제 감지 – 모니터링 기반 접근

초기 구현 후, 폴리곤 렌더링 시 특정 구역 조회에서 응답 지연이 크게 튀는 현상을 발견했다.

- 모니터링 로그와 쿼리 플랜을 확인해 보니:
  - 일부 조회에서 MySQL이 테이블 풀 스캔을 수행

  - 인덱스가 있어야 할 컬럼에 인덱스가 누락되어 있고

  - WHERE 조건이 인덱스를 활용하기 어렵게 작성된 부분이 있었다.

단순히 “인덱스를 추가했다”가 아니라,
쿼리 플랜을 보고 실제로 어떤 부분이 병목인지 확인한 뒤 재설계하는 방향으로 접근했다.

## 구현 내용과 결과

4-1. 인덱스/쿼리 플랜 재설계로 응답 시간 개선
문제: 폴리곤 렌더링용 데이터 조회에서 풀 스캔이 발생해 응답 시간이 길어짐.

해결:

금연/흡연 구역 데이터에 대해 위도/경도 복합 인덱스와 외래키(FK) 인덱스를 다시 설계

쿼리 조건을 인덱스를 잘 탈 수 있도록 정리하고, 쿼리 플랜(EXPLAIN)으로 인덱스 사용 여부를 확인

문제 구간을 중심으로 쿼리를 리팩터링

결과 (대표 수치):
| 구분 | 적용 전 응답 속도 | 적용 후 응답 속도 |
| ------------- | ---------- | ---------- |
| 일반 구역 조회 API | 180~185ms | 23~25ms |
| 아파트 구역 조회 API | 40~44ms | 8~11ms |

이때 중요한 것은 “84.7% 개선”이라는 숫자 자체보다,
풀 스캔 병목 파악 → 인덱스/쿼리 플랜 재설계 → 지연 감소라는 문제 해결 프로세스를 경험한 것이다.

4-2. 캐싱 전략으로 API 호출량 절감
도메인 특성: 흡연/금연 구역 데이터는 자주 바뀌지 않는 정적에 가까운 데이터다.

이를 활용해 다음과 같은 2중 캐싱 전략을 설계했다.

클라이언트 측 IndexedDB 캐시 (약 30일 보존)

브라우저/프론트에서 구역 데이터를 IndexedDB에 저장해, 재방문 시 네트워크 요청 없이 지도 렌더링 가능.

TanStack Query 캐시 (staleTime 1시간)

같은 세션/단기간 내 반복 조회에 대해 API 요청을 합리적으로 줄이기 위해 사용.

추가로:

월말 배치 스케줄러로 서버 측 데이터 변경분만 차분 업데이트

구역 데이터에 버전 정보를 부여해, 클라이언트 쪽에서 버전이 달라졌을 때만 캐시를 무효화하는 구조를 적용했다.

이 전략으로 반복 방문 시 API 호출량을 약 95%까지 줄이면서,
데이터 최신성도 유지할 수 있었다.

여기서 강조할 부분은 “95% 절감”이라는 숫자 그 자체보다,
캐시 무효화를 월말 배치 + 버전 비교로 설계했다는 점이다

## 팀 협업과 내가 맡은 역할

### 백엔드 리더(PL)로서의 역할

- 6인 팀에서 백엔드 리더(PL)로 참여해:

- 금연/흡연 구역 도메인 이해를 팀원과 함께 정리

- DB 스키마 설계 및 MySQL 인덱스 전략 수립

- 가장 많이 호출되는 주요 API를 설계·구현

- 성능 문제를 직접 모니터링하고,
  로그/쿼리 플랜 기반으로 병목을 찾아내 해결하는 역할을 담당했다.

### 데이터 수집 툴 개발과 협업 경험

- 친구들이 현장 답사로 모은 데이터를 잃지 않도록:
  - 좌표/메타데이터 입력용 수집 웹사이트와

  - 해당 데이터를 CleanBreath 서비스 DB에 적재하는 백엔드를 구현했다.

- 이 경험을 통해:
  - “현장의 도메인 지식과 노력”을 기술로 어떻게 받쳐 줄지

  - 팀 커뮤니케이션이 설계 품질에 어떻게 영향을 미치는지
    를 몸으로 느낄 수 있었다.

## 한계점과 보완 방향

### 데이터 커버리지 한계

현재 CleanBreath는 안양시 특정 구역에 집중된 데이터를 다루고 있어,
전국/광역 서비스로 보기엔 데이터 커버리지가 부족하다.

#### 보완 방향:

수집 툴을 일반 사용자/지자체도 쉽게 사용할 수 있는 형태로 확장해,
크라우드소싱 기반 데이터 수집 모델을 도입할 수 있다.

지자체·보건소와 협업해 공식 데이터와 현장 데이터를 병합하는 전략도 고려.

### 인프라/운영 측면

현재 백엔드는 EC2 t4g.micro 24/7 운영 구조로 돌아가고 있다.

트래픽이 매우 크지는 않지만, 인스턴스 비용 + VPC 비용이 계속 나가는 구조라
장기적으로는 비용 최적화가 필요하다.

#### 보완 방향(생각해볼 수 있는 옵션):

트래픽 패턴을 고려해 인스턴스 타입 다운사이징 또는 예약 인스턴스/세이빙 플랜 검토

지도 조회 API 일부를 캐시 서버/서버리스로 이관해, EC2 부하를 줄이는 하이브리드 구조

미사용 리소스(NAT Gateway, 사용 안 하는 EIP, VPC Endpoint 등) 정리로 VPC 비용 절감

### 실시간성/알림 기능 부족

현재는 지도 조회 중심 서비스라,
실시간 금연단속 정보, 신고 기능, 알림(예: “이 구역 금연으로 변경됨”) 등은 구현되어 있지 않다.

#### 보완 방향:

특정 구역의 정책 변경/단속 정보를 관리자가 등록하면,
해당 구역을 즐겨찾기한 사용자에게 알림을 주는 기능

시민 신고/제보 기능을 추가해, 데이터 최신성을 사용자 커뮤니티로부터 확보
